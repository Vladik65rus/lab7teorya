### 1.1 Общая часть

1. Установить Clang, LLVM, `opt` и Graphviz.
2. Скомпилировать простой C-файл и получить:
   - абстрактное синтаксическое дерево (AST);
   - промежуточное представление LLVM IR.
3. Применить базовую комплексную оптимизацию (`-O2`).
4. Построить граф потока управления (CFG) для оптимизированной программы.
5. Проанализировать результат и сделать выводы.

### 1.2 Индивидуальное задание (вариант 2.10)

Исходный код (`main.cpp`):

```cpp
#include <cstdio>

inline int square(int x) {
    return x * x;
}

int main() {
    int a = 5;
    int b = square(a);
    return b;
}
```

Требуется:

1. Получить IR для `-O0`.
2. Получить IR для `-O2`. Встроилась ли функция?
3. Применить `-always-inline` и сравнить с предыдущими оптимизациями.
4. Построить CFG до и после.
5. Сделать вывод об условиях встраивания функций в LLVM.

### 2.4 Генерация LLVM IR
<img width="1919" height="1079" alt="image" src="https://github.com/user-attachments/assets/838e22d6-693b-433b-bffd-4c4133332f1a" />

```bash

clang -O2 -S -emit-llvm main.c -o main_O2.ll


```

В файле `main_O0.ll`:

- значения читаются и пишутся через `load`/`store`;
- `square` остаётся отдельной функцией и вызывается через `call`.

### 2.5 Оптимизация IR (`-O2`)

При `-O2` применяется более 30 проходов, ключевые из них:
- `mem2reg` — перевод переменных из памяти в виртуальные регистры (SSA-форма);
- `inline` — встраивание небольших функций;
- `sccp` / `constprop` — свёртка констант;
- `instcombine` — упрощение арифметических инструкций;
- `simplifycfg` — упрощение графа потока управления;
- `dce`, `globaldce` — удаление мёртвого кода.

```bash

```

После оптимизаций:
- `alloca`/`load`/`store` удалены;
- функция `square` встроена в `main` и затем удалена как мёртвый код;
- от `main` остался только вызов `printf` с константой 25.

### 2.7 Выводы по общей части

- LLVM предоставляет богатый набор инструментов для анализа и оптимизации, причём

- Промежуточное представление и CFG удобны для написания компиляторных

## 3. Индивидуальное задание (вариант 2.10)

### 3.1 Исходный код

```cpp
#include <cstdio>

inline int square(int x) {
    return x * x;
}

int main() {
    int a = 5;
    int b = square(a);
    return b;
}
```

> В качестве C++ файла, так как `inline` в C требует `static inline` или отдельной
> внешней декларации для одиночного определения.

### 3.2 AST

Команда:
```bash

```

Ключевые узлы (сокращённо):
```
|-FunctionDecl  square 'int (int)' inline
| |-ParmVarDecl used x 'int'
| `-CompoundStmt
```

`inline` сохраняется в AST как атрибут декларации, но это только подсказка

### 3.3 LLVM IR при `-O0`
<img width="1919" height="116" alt="image" src="https://github.com/user-attachments/assets/4aae8143-cc83-47be-8381-daf0f10c2ef9" />

```bash
clang -O0 -S -emit-llvm main.cpp -o main_O0.ll


```

```llvm
; Function Attrs: noinline nounwind optnone uwtable
define linkonce_odr dso_local i32 @_Z6squarei(i32 noundef %0) #0 {
  %2 = alloca i32, align 4
  store i32 %0, i32* %2, align 4
  %3 = load i32, i32* %2, align 4
  %4 = load i32, i32* %2, align 4
  %5 = mul nsw i32 %3, %4
  ret i32 %5
}

; Function Attrs: noinline nounwind optnone uwtable
define dso_local i32 @main() #0 {
  %1 = alloca i32, align 4
  %2 = alloca i32, align 4
  %3 = alloca i32, align 4
  store i32 0, i32* %1, align 4
  store i32 5, i32* %2, align 4
  %4 = load i32, i32* %2, align 4
  %5 = call i32 @_Z6squarei(i32 noundef %4)
  store i32 %5, i32* %3, align 4
  %6 = load i32, i32* %3, align 4
  ret i32 %6
}
```

Наблюдения:
- атрибут `optnone noinline` запрещает встраивание;
- `square` имеет linkage `linkonce_odr` (стандартное поведение inline-функций в C++);
- все переменные размещены через `alloca`, обмен данными через `load`/`store`;
- вызов `square` присутствует как полноценный `call`.

### 3.4 LLVM IR при `-O2`
<img width="1919" height="129" alt="image" src="https://github.com/user-attachments/assets/3812ade5-345a-4db1-9db0-2663ba5a6d12" />

```bash
clang -O2 -S -emit-llvm main.cpp -o main_O2.ll


```

```llvm
; Function Attrs: mustprogress nofree norecurse nosync nounwind
;                 readnone uwtable willreturn
define dso_local i32 @main() local_unnamed_addr #0 {
  ret i32 25
}
```

Что произошло:
- `mem2reg` устранил `alloca`/`load`/`store`, IR переведён в SSA-форму;
- `inline` встроил `square` в `main` (функция короткая, вызывается один раз,
  имеет linkonce_odr linkage и пользовательский `inline`);
- `sccp` + `instcombine` свернули `5 * 5` в константу `25`;
- определение `square` удалено как мёртвый код (`globaldce`);
- `main` получила атрибуты `readnone willreturn` — LLVM формально доказал
  отсутствие побочных эффектов.

**Ответ:** функция `square` встроена при `-O2` и удалена из IR полностью.

### 3.5 Применение `-always-inline`

Изменим объявление `square`:

```cpp
__attribute__((always_inline)) inline int square(int x) {
    return x * x;
}
```
<img width="1919" height="125" alt="image" src="https://github.com/user-attachments/assets/7d1ad12b-a67a-4e11-848a-5949352a8ae7" />

```bash
clang -O0 -S -emit-llvm main.cpp -o main_O0.ll
opt -passes=always-inline -S main_O0.ll -o main_ainl.ll


```

В `main_ainl.ll`:

```llvm
define dso_local i32 @main() #0 {
  %1 = alloca i32, align 4
  %2 = alloca i32, align 4
  %3 = alloca i32, align 4
  store i32 0, i32* %1, align 4
  store i32 5, i32* %2, align 4
  %4 = load i32, i32* %2, align 4
  ; --- встроенное тело square ---
  %5 = alloca i32, align 4
  store i32 %4, i32* %5, align 4
  %6 = load i32, i32* %5, align 4
  %7 = load i32, i32* %5, align 4
  %8 = mul nsw i32 %6, %7
  ; --- конец встроенного тела ---
  store i32 %8, i32* %3, align 4
  %9 = load i32, i32* %3, align 4
  ret i32 %9
}
```

Сравнение трёх вариантов:

| Режим | Встраивание | `alloca`/`load`/`store` | Свёртка `5*5 = 25` |
|---|---|---|---|
| `-O0` | нет | да | нет |
| `-O0 + -always-inline` | да | да | нет |
| `-O2` | да | нет | да |

`-always-inline` выполняет **только** подстановку тела. Для полной оптимизации
нужны сопровождающие проходы (`mem2reg`, `sccp`, `instcombine`, `simplifycfg`, `dce`).

### 3.6 CFG до и после

```bash
clang -O0 -S -emit-llvm main.cpp -o main_O0.ll
opt -dot-cfg -disable-output main_O0.ll
dot -Tpng .main.dot         -o cfg_main_O0.png
dot -Tpng ._Z6squarei.dot   -o cfg_square_O0.png

clang -O2 -S -emit-llvm main.cpp -o main_O2.ll
opt -dot-cfg -disable-output main_O2.ll
dot -Tpng .main.dot -o cfg_main_O2.png
```

CFG функции `square` при `-O0` — один базовый блок (`alloca`, два `load`, `mul`, `ret`).
CFG функции `main` при `-O0` — один базовый блок с `alloca` × 3, серией `store`/`load`,
вызовом `call @_Z6squarei` и `ret`.
CFG функции `main` при `-O2` — один базовый блок с единственной инструкцией
`ret i32 25`. CFG функции `square` при `-O2` не существует (определение удалено).

Структура графа не изменилась (один блок входа = блок выхода), но содержимое
блока радикально сократилось.

### 3.7 Выводы по индивидуальному заданию

1. Ключевое слово `inline` в C/C++ — рекомендация, а не приказ; решение
   принимает inliner на уровне LLVM IR.
2. При `-O0` функции получают атрибуты `noinline optnone`, и встраивание
   полностью отключено.
3. При `-O2` маленькая нерекурсивная функция, вызываемая один или несколько
   раз, почти всегда встраивается, особенно если её linkage допускает удаление
   (`static`, `internal`, `linkonce_odr`).
4. Гарантированное встраивание обеспечивает `__attribute__((always_inline))`;
   соответствующий pass работает даже на `-O0`.
5. `-always-inline` подставляет только тело — без сопутствующих pass-ов
   реальной выгоды (свёртка констант, регистровая форма) не будет.
6. После встраивания и оптимизаций определение исходной функции может быть
   удалено полностью (в нашем примере `square` исчезает из `main_O2.ll`).
7. Встраивание не всегда выгодно: большие, рекурсивные функции и функции с
   виртуальными вызовами inliner оставляет нетронутыми, чтобы не «раздуть»
   код и не ухудшить локальность инструкционного кэша.

## 4. Ответы на контрольные вопросы

**1. Что такое Clang и какова его роль в процессе компиляции программ?**
Clang — фронтенд компилятора для языков семейства C (C, C++, Objective-C). Он выполняет
лексический, синтаксический и семантический анализ исходного кода, строит AST и
порождает промежуточное представление LLVM IR, которое затем обрабатывает LLVM-бэкенд.
Clang отвечает за всё, что специфично для языка; LLVM — за всё, что универсально
(оптимизации, генерация машинного кода).

**2. Что такое LLVM и как он используется в современных компиляторах?**
LLVM — модульный набор библиотек и инструментов для построения компиляторов:
оптимизатор, генератор кода, JIT, линкер, отладочные утилиты. Его ключевая идея —
универсальное промежуточное представление (LLVM IR), независимое и от языка
источника, и от целевой архитектуры. Современные компиляторы (Clang, Rust, Swift,
Julia, Kotlin/Native, Zig и др.) используют LLVM как общий бэкенд: фронтенд каждого
языка генерирует LLVM IR, а оптимизации и кодогенерация выполняются унифицированно.

**3. Чем отличается AST от LLVM IR?**
AST — древовидное представление, близкое к исходному коду: в нём сохранены типы,
имена переменных, конструкции языка (`if`, `for`, `switch`), синтаксический сахар.
LLVM IR — низкоуровневое линейное представление с базовыми блоками, типизированными
SSA-регистрами и небольшим набором инструкций (`add`, `load`, `store`, `br`, `call`
и т. п.). AST — для семантического анализа, IR — для оптимизаций и кодогенерации.

**4. Для чего необходимо промежуточное представление в процессе компиляции?**
IR разделяет компилятор на фронтенды и бэкенды: один и тот же оптимизатор и
кодогенератор работают для любого языка и любой целевой архитектуры. Это даёт:
переиспользование кода оптимизаций; единый язык для написания компиляторных
проходов; возможность встраивания между языками (LTO); более удобный анализ —
SSA, базовые блоки и регулярная форма инструкций гораздо проще для алгоритмов,
чем произвольный AST или текст программы.

**5. Что делает инструкция `alloca` в LLVM IR и зачем она нужна?**
`alloca` резервирует память в стеке текущей функции и возвращает указатель на неё.
Используется как «канонический контейнер» для адресуемых локальных переменных:
clang при `-O0` всегда кладёт каждую переменную в `alloca`, а затем работает с ней
через `load`/`store`. После прохода `mem2reg` большинство `alloca` исчезают —
переменные превращаются в SSA-регистры. Остаются только те `alloca`, у которых
есть «реальные» причины существовать (адрес берётся, тип не помещается в регистр и т. п.).

**6. Зачем нужна оптимизация кода в компиляторе и какие цели она преследует?**
Цели оптимизации: уменьшить время исполнения, уменьшить размер кода и потребление
памяти, улучшить использование кэша и предсказателя переходов, иногда — снизить
энергопотребление. Оптимизатор устраняет избыточные вычисления (CSE, GVN),
сворачивает константы, встраивает функции, разворачивает циклы, удаляет мёртвый
код, переупорядочивает инструкции для конвейера. Без оптимизатора программа на
языках высокого уровня была бы в разы медленнее эквивалентной программы на
ассемблере.

**7. Что такое SSA-форма и почему она важна при оптимизации программ?**
SSA (Static Single Assignment) — форма IR, в которой каждая переменная присваивается
ровно один раз. Если в исходном коде одно имя присваивается многократно, в SSA
оно превращается в несколько разных версий; для слияния значений на сходящихся
ветвях вводится псевдоинструкция `phi`. SSA важна потому, что в ней use-def
зависимости тривиальны (от использования сразу видно единственное место
определения). Это упрощает почти все алгоритмы оптимизации: constant
propagation, GVN, DCE, register allocation работают в SSA в разы быстрее и
точнее.

**8. Что такое граф потока управления (CFG) и как он помогает анализировать поведение программы?**
CFG — ориентированный граф, узлы которого — базовые блоки (последовательности
инструкций без ветвлений внутри), а рёбра — возможные переходы между ними.
По CFG строятся все классические анализы: достижимость кода, доминирование,
циклы, потоки данных, live-variable analysis. Без CFG невозможно сказать, какие
инструкции исполняются последовательно, какие — условно, и какие — никогда.

**9. Как устроено представление арифметических операций в LLVM IR (сложение, умножение)?**
Каждая арифметическая операция — отдельная типизированная инструкция:
`add`, `sub`, `mul`, `udiv`/`sdiv`, `urem`/`srem` для целых, `fadd`, `fsub`,
`fmul`, `fdiv` для вещественных. Целочисленные операции имеют флаги переполнения
(`nsw` — no signed wrap, `nuw` — no unsigned wrap), вещественные — флаги быстрой
арифметики (`fast`, `nnan`, `ninf` и т. д.). Все операции принимают значения
одного типа и возвращают значение того же типа. Например, `%5 = mul nsw i32 %3, %4`
выполняет целочисленное знаковое умножение 32-битных операндов.

**10. Почему функции в LLVM IR обычно представляют собой отдельные единицы анализа и оптимизации?**
Большинство оптимизаций — внутрипроцедурные: CFG, доминирование, SSA, GVN,
mem2reg, simplifycfg работают в пределах одной функции. Это делает их
быстрыми и параллелизуемыми (разные функции можно оптимизировать
независимо). Межпроцедурные оптимизации (inlining, IPA, IPSCCP, dead function
elimination) выполняются отдельным набором проходов поверх внутрипроцедурных
и обычно применяются после.

**11. Что происходит с функцией в LLVM IR, если она вызывается один раз и очень короткая?**
При `-O2` такая функция почти наверняка будет встроена в место вызова, а затем
её определение будет удалено (если linkage позволяет). Это даёт двойную выгоду:
исчезает стоимость вызова, и тело функции участвует в оптимизациях вместе с
кодом вызывающей функции (свёртка констант, упрощение и т. д.). В нашем примере
`square` при `-O2` исчезает целиком, а `main` сводится к `ret i32 25`.

**12. Какие преимущества даёт использование IR и CFG для автоматических оптимизаций по сравнению с анализом исходного текста на C?**
- IR унифицирован: одна и та же оптимизация работает для C, C++, Rust, Swift и др.
- IR проще: десятки инструкций вместо сотен языковых конструкций.
- SSA-форма даёт прямые use-def связи; в исходном тексте их пришлось бы
  восстанавливать.
- CFG даёт чёткую структуру управления; на исходном тексте `goto`, `switch`,
  `break`, исключения и макросы делают анализ управляющего потока сложным.
- IR независим от целевой архитектуры — оптимизации не привязаны к ассемблеру.
- Над IR удобно писать многократно применяемые проходы (`-mem2reg`, `-instcombine`),
  чего почти невозможно достичь, работая с текстом исходного кода.
